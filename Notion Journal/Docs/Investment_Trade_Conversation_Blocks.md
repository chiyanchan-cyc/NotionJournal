# Investment Trade Conversation Blocks

Purpose: make every trade a living conversation inside Notion Journal blocks, while preserving enough structure for Codex heartbeat, search, CloudKit sync, and future RAG.

## Core Rule

Trades are not only tables. A trade is:

- a thesis card,
- a watch list table,
- a target KPI table,
- a set of real transaction cards if money is deployed,
- a movement log,
- and a conversation timeline made of normal Notion Journal blocks.

The conversation timeline should use the existing `nj_block` / `nj_note_block` system so it syncs through CloudKit and stays readable in the journal.

## Trade Page Top Blocks

Every trade page should start with four collapsible blocks before the conversation timeline.

These four blocks are pinned operating blocks:

- Watch List
- Target KPI
- Actual Trades
- Movement Log

They stay above the conversation regardless of how long the conversation becomes.

### 1. Watch List

The watch list holds the target list of stocks, ETFs, crypto, commodities, or other instruments attached to the trade thesis.

Columns:

- `trade_id`
- `symbol`
- `instrument_name`
- `market`
- `reason_to_watch`
- `target_zone`
- `status`
- `last_price`
- `daily_change`
- `as_of`
- `source`

The daily heartbeat should refresh each watch-list item and record whether it is:

- moving toward target,
- moving away from target,
- neutral,
- invalidating the premise,
- or requiring deeper research.

### 2. Target KPI

Target KPI holds the measurable conditions that decide whether we should enter, wait, reduce confidence, or invalidate.

Columns:

- `trade_id`
- `kpi_name`
- `target_value`
- `current_value`
- `direction_needed`
- `action_rule`
- `status`
- `as_of`
- `source`

Examples:

- Mag 7 relative strength holds versus QQQ.
- NVDA guidance supports the AI capex cycle.
- SaaS revenue / FCF revisions stabilize while price capitulates.
- USDCNH stays stable while tariff language softens.

### 3. Actual Trades

Actual Trades records only real user-confirmed transactions. Codex may suggest an action request, but it must not create a real trade row unless the user confirms execution.

Columns:

- `transaction_card_id`
- `trade_id`
- `symbol`
- `side`
- `quantity`
- `entry_price`
- `entry_date`
- `position_status`
- `linked_thesis_block_id`
- `notes`

### 4. Movement Log

Movement Log records daily movement of watch-list symbols and the interpretation against the thesis.

Columns:

- `movement_id`
- `date_key`
- `trade_id`
- `symbol`
- `price`
- `daily_change`
- `relative_move`
- `movement_read`
- `impact_on_thesis`
- `next_step`
- `source`

This is the most important daily record because it lets us later ask whether the trade idea was working before we acted.

## Conversation Order

The trade conversation timeline is newest first.

Reason:

- active trade questions need to be visible immediately,
- Codex heartbeat should pick up the newest pending question first unless priority says otherwise,
- older context remains available below without pushing the operating blocks away.

The four pinned operating blocks are not part of the newest-first sort. They always stay at the top.

## Trade Lifecycle

Each trade needs a monitoring status:

- `active`: heartbeat refreshes watch list, KPIs, movement log, and pending questions.
- `paused`: heartbeat answers explicit questions, but does not run daily market movement monitoring.
- `ended`: heartbeat stops monitoring the trade. The full history remains readable and searchable.
- `archived`: hidden from active trade views unless explicitly included.

When a trade ends:

1. Add a final `verdict` block.
2. Mark the trade status `ended`.
3. Stop daily watch-list and KPI refresh.
4. Keep actual trade records and movement history immutable except for corrections.
5. Keep conversation blocks available for RAG and later postmortem.

## Block Roles

Each trade conversation block needs an author role:

- `user`: written by the user.
- `codex`: written by Codex.
- `system`: generated by automation or import.

Each block also needs an interaction kind:

- `note`: ordinary trade note.
- `question`: a question the user wants Codex to answer.
- `answer`: Codex answer to a question.
- `thesis_update`: change to the trade premise.
- `risk_update`: new invalidation or risk.
- `action_request`: possible action that still needs confirmation.
- `transaction_ref`: points to a real transaction card.
- `verdict`: weekly or post-trade judgment.

## Payload Shape

Store this inside the block `payload_json` under a `trade_conversation` key. Keep the usual text/clip/proton body intact so the block remains readable.

```json
{
  "sections": {
    "clip": {
      "type": "text",
      "data": {
        "body": "Readable block text goes here."
      }
    }
  },
  "trade_conversation": {
    "version": 1,
    "trade_id": "dream-girl-mag7",
    "author_role": "user",
    "interaction_kind": "question",
    "question_status": "pending",
    "question_id": "q_20260427_001",
    "parent_question_id": "",
    "source_tab": "codex:investment-macro-heartbeat",
    "needs_codex": true,
    "priority": "normal",
    "created_in_context": "Trade Thesis",
    "linked_transaction_card_id": "",
    "source_links": [],
    "as_of": "2026-04-27T20:30:00+08:00"
  }
}
```

## Question Block

A question block is a normal journal block where:

- `interaction_kind = "question"`
- `author_role = "user"` or `system`
- `needs_codex = true`
- `question_status = "pending"`

Codex heartbeat should treat pending question blocks as work items.

Question statuses:

- `pending`
- `running`
- `answered`
- `blocked`
- `stale`

When Codex answers, it should create a new block:

- `interaction_kind = "answer"`
- `author_role = "codex"`
- `parent_question_id = <question_id>`
- `question_status = "answered"`

Then it should update the original question block to `question_status = "answered"` if the app supports safe payload update.

## Visible UI Labels

Each trade conversation block should show small metadata chips:

- `Me` or `Codex`
- `Question`, `Answer`, `Thesis`, `Risk`, `Action`, `Transaction`, or `Verdict`
- status chip for questions: `Pending`, `Running`, `Answered`, `Blocked`

Question blocks should have a clear visual affordance:

- left accent line,
- question mark icon,
- pending status chip,
- optional `Ask Codex` action.

Codex answer blocks should be visually distinct but not loud:

- Codex chip,
- answer icon,
- source count if sources exist,
- confidence or uncertainty note when useful.

## Tags

Use `tag_json` and `nj_block_tag` for retrieval tags:

- `me.finance`
- `investment`
- `trade.<trade_id>`
- `market.us`, `market.china_hk`, `market.japan`, or `market.commodity_crypto`
- `#TRADE_CONVERSATION`
- `#CODEX_QUESTION` for question blocks
- `#CODEX_ANSWER` for answer blocks

Example:

```json
[
  "me.finance",
  "investment",
  "trade.dream-girl-mag7",
  "market.us",
  "#TRADE_CONVERSATION",
  "#CODEX_QUESTION"
]
```

## Heartbeat Behavior

Each heartbeat tab should watch only the trade IDs assigned to it.

Daily loop:

1. Find pending question blocks for its trade ID.
2. Read the linked thesis card and recent conversation blocks.
3. Answer the question in a new Codex block.
4. Update the original question status.
5. If the answer changes the trade premise, add a separate `thesis_update` block.
6. If the answer implies action, add an `action_request` block, not a real transaction.

Codex should never create a real transaction card unless the user explicitly confirms a real transaction happened.

## Trade Memory Loading

To reduce token cost, heartbeat should load:

1. trade thesis card,
2. latest trade capsule,
3. unanswered question blocks,
4. recent 10 to 20 conversation blocks,
5. older blocks only when needed.

The full conversation remains in Notion Journal and CK, but Codex does not need to prefill it every time.
